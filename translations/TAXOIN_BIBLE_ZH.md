## 序言

起初，有工作量证明。矿工燃烧电力计算哈希值，网络用代币奖励他们。
这项工作对安全有用，但对人类无用。

Taxoin 的创建是为了解决一个根本问题：

> **用有用的贡献取代无用的计算。**

不是"一个 CPU 一票"（中本聪，2008）。
而是 **一个人，一次贡献，一份价值。**

## 第一部分：什么是 Taxoin？

### I.1 定义

**Taxoin (Ⓣ)** 是对 Taxoin 网络**已证明贡献**的单位。
它不能通过挖矿获得，不能铸造，也不能凭空产生。

每个进入网络的 Ⓣ 都通过以下两个入口之一：

| 入口 | 机制 | 最大供应量 |
|------|------|-----------|
| **创世 (Genesis)** | 人性证明 (Proof of Personhood) — 一人一创世 | 21,000,000 Ⓣ (420,000 人 × 50Ⓣ) |
| **服务 (Service)** | 贡献证明 (Proof of Contribution) — 提供真实服务 | 无限（受实际经济活动限制） |

### I.2 核心原则

> **没有相应的贡献，就没有 Taxoin。**

这是基本法则。每个代币都有对应的：
- 一个独特的人类（创世），或
- 一个经过验证的服务（贡献）

没有矿工从电力中提取代币。没有质押者赚取被动收入。
只有 **某人认为有价值并愿意付费的工作。**

### I.3 非原则（Taxoin 不是什么）

| ❌ 不是这个 | ✅ 而是这个 |
|------------|------------|
| 投资资产 | 服务交换媒介 |
| 价值储存 | 贡献单位 |
| 挖矿奖励 | 服务报酬 |
| 被动收入 | 主动工作收入 |
| 投机代币 | 已完成工作的证明 |

## 第二部分：创世 (Genesis) — 人性证明 (Proof of Personhood)

### II.1 引导问题

服务经济不能从零开始：
- 没有人有代币支付服务费用
- 没有人能在不提供服务的情况下赚取代币
- 死锁

### II.2 解决方案：一人一创世

Taxoin 不是向"矿工"或"投资者"分发创世代币，而是向经过验证者集验证的**独特人类**分发。

```
全局参数：
  GENESIS_REWARD     = 50 Ⓣ  （每个独特的人）
  MAX_GENESIS_SUPPLY = 21,000,000 Ⓣ
  MAX_PARTICIPANTS   = 420,000  （21M / 50）
```

### II.3 创世仪式

```
  ┌──────────────────────────────────────────────────────────────┐
  │                  创世仪式 (GENESIS CEREMONY)                   │
  │                                                              │
  │  1. 人类生成密钥对 (secp256k1)                                │
  │     → 地址: 0xdeadbeef...                                    │
  │                                                              │
  │  2. 人类联系验证者（亲自/视频通话/通过可信推荐人）               │
  │     → 证明他们是独特的人类                                    │
  │     → 用密钥签署消息："我是真实的"                              │
  │                                                              │
  │  3. 验证者验证：                                              │
  │     → 签名与声明的地址匹配                                    │
  │     → 该人类尚未在创世注册表中                                │
  │     → 该人类看起来是真实的独特人类                              │
  │                                                              │
  │  4. 验证者签署证明：                                          │
  │     "0xval1 证明 0xdeadbeef 是一个独特的人类"                  │
  │                                                              │
  │  5. 需要 3/7 验证者证明 → 创世批准                             │
  │     → +50 Ⓣ 记入 0xdeadbeef                                 │
  │     → 地址加入创世注册表（不可变）                              │
  │                                                              │
  └──────────────────────────────────────────────────────────────┘
```

### II.4 Sybil 抵抗

Sybil 攻击（一个人创建 1000 个钱包）通过以下方式防止：

| 层级 | 保护 |
|-------|-----------|
| **人性证明 (Proof of Personhood)** | 验证者验证人，而不是钱包 |
| **3/7 证明** | 需要 3 个独立验证者确认 |
| **不可变注册表** | 每个地址只能创世一次 |
| **声誉** | 批准 Sybil 的验证者失去声誉 |

这并不完美。它需要社会信任。但这是已知的唯一无需以下方式即可公平分发代币的方法：
- 工作量证明 (PoW)（ASIC 中心化）
- 权益证明 (PoS)（富者愈富）
- 首次代币发行 (ICO)（内部人优势）
- 空投（Sybil 农场）

### II.5 创世上限

```
MAX_GENESIS_SUPPLY = 21,000,000 Ⓣ
```

这是创世代币的**绝对上限**。一旦 420,000 人通过认证，就不能再创建创世代币。

创世耗尽后，流通中的所有新 Ⓣ 完全来自服务交易（一个人向另一个人支付服务费用）。

## 第三部分：服务经济 — 贡献证明 (Proof of Contribution)

### III.1 什么是服务？

服务是一个网络参与者向另一个参与者提供的任何贡献。
服务在**链上注册**并由**提供者定价**。

示例：
```
  服务类型             提供者              价格
  ─────────────────────────────────────────────────
  SMS 网关            Alice 的手机        0.1 Ⓣ/SMS
  GPU 计算            Bob 的矿场          5.0 Ⓣ/小时
  API 访问            Carol 的模型        0.5 Ⓣ/次
  乘车（出租车）       Dave 的车           3.0 Ⓣ/公里
  文件存储            Eve 的服务器         0.01 Ⓣ/MB/月
  CPU 周期            Frank 的桌面         0.001 Ⓣ/CPU-小时
  带宽中继            Grace 的路由器       0.05 Ⓣ/GB
  人工审核            Hank 的专业知识      10.0 Ⓣ/次审核
```

服务可以是什么没有限制。市场决定哪些服务有价值以及以什么价格。

### III.2 服务注册表 (Service Registry)

每项服务都作为 git 提交在链上注册：

```python
@dataclass
class ServiceRegistration:
    provider: str              # 地址 (0x...)
    service_type: str          # "sms", "gpu", "taxi", ...
    price_per_unit: float      # Ⓣ 每单位
    description: str           # 人类可读
    endpoint: str              # 如何调用 (URL, contact, API)
    attested_by: list[str]     # 已验证的验证者
    rating: float = 0.0
    total_tx: int = 0
    created_at: float
```

服务注册表存储在 git notes（现有基础设施）中。
任何人都可以查询："显示所有评分 > 4.0 的 SMS 网关。"

验证者在批准注册前验证服务是否真实：
- 对于 SMS 网关：发送测试 SMS，验证投递情况
- 对于 GPU 计算：运行基准测试，验证输出
- 对于出租车行程：验证驾驶执照和车辆

### III.3 定价

提供者设定价格。市场调节价格。

```
  高价 + 差服务  →  没有客户 → 提供者降低价格
  低价 + 好服务  →  许多客户 → 提供者变得有声誉
  欺诈（收钱，不提供服务） → 争议 → 验证者仲裁
                                    → 声誉处罚
                                    → 可能的封禁
```

没有中央机构设定价格。没有算法调整费用。
纯粹的自由市场。

### III.4 发现

消费者通过注册表发现服务：

```
  taxoin service list --type sms --min-rating 4.0
  → 0xalice: 0.1 Ⓣ/SMS, 评分 4.8 (942 笔交易)
  → 0xbob:   0.15 Ⓣ/SMS, 评分 4.2 (312 笔交易)
```

## 第四部分：交易生命周期 — 共同证明 (Mutual Attestation)

### IV.1 核心问题

> 区块链如何知道服务确实已交付？

在传统区块链中，这被称为**预言机问题 (Oracle Problem)**。
Taxoin 的回答：**共同证明 (Mutual Attestation)** — 双方签署。

### IV.2 共同证明交易 (AttestedTransaction)

```python
@dataclass
class AttestedTransaction:
    tx_id: str
    consumer: str              # 谁支付
    provider: str              # 谁提供
    service_ref: str           # 链接到 ServiceRegistration
    amount: float              # 本次交易 Ⓣ
    consumer_sig: str          # "我已收到服务" (ECDSA)
    provider_sig: str          # "我已提供服务" (ECDSA)
    timestamp: float
    description: str           # 可选的订单详情
```

**交易仅在双方签名都存在时才有效。**

### IV.3 生命周期

```
  ┌─────────────────────────────────────────────────────────────────┐
  │                   交易生命周期 (TRANSACTION LIFECYCLE)           │
  │                                                                 │
  │  1. 消费者浏览注册表                                            │
  │     → 找到他们需要的服务                                        │
  │     → 检查价格、评分、提供者声誉                                │
  │                                                                 │
  │  2. 消费者发送请求                                              │
  │     → 调用服务端点（链下）                                      │
  │     → "我要发送 1 条短信到 +79123456789"                       │
  │                                                                 │
  │  3. 提供者执行服务（链下）                                      │
  │     → 发送 SMS                                                  │
  │     → 等待投递确认                                              │
  │                                                                 │
  │  4. 共同证明 (Mutual Attestation)                               │
  │     → 消费者签署："我确认，短信已收到"                          │
  │     → 提供者签署："我确认，短信已发送"                          │
  │     → 双方签名 → AttestedTransaction 创建                       │
  │                                                                 │
  │  5. 余额转移（通过共识在链上）                                  │
  │     → 验证者验证双方签名 (ECDSA)                                │
  │     → 验证者检查消费者有足够余额                                │
  │     → 5/7 验证者确认 → 交易提交                                 │
  │     → -0.1 Ⓣ 消费者, +0.1 Ⓣ 提供者                            │
  │     → 双方 +1 声誉                                             │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

### IV.4 余额保留 (Balance Hold)

在服务执行期间，消费者的代币被**保留**（预留）但尚未支出：

```
  消费者余额 之前:  100.0 Ⓣ
    ├── 可用:           99.9 Ⓣ
    ├── 保留中:          0.1 Ⓣ  （SMS 进行中）
    └── 锁定:            0.0 Ⓣ

  共同证明后 → 保留的已支出，记入提供者
  10 区块超时后 → 保留的退还给消费者（服务失败）
```

### IV.5 争议解决 (Dispute Resolution)

如果一方作弊怎么办？

| 场景 | 解决 |
|----------|-----------|
| 消费者签署，提供者不签署 | 超时 → 保留的退还给消费者 |
| 提供者交付，消费者拒绝签署 | 提供者提出争议 → 验证者审查证据 → 5/7 规则 |
| 双方签署，但服务是假的 | 验证者检测模式 → 争议 → 声誉处罚 |
| 提供者收钱，不交付 | 消费者提出争议 → 验证者裁决 → 资金退还 |

验证者是最终仲裁者。有 7 个验证者和 5/7 法定人数，
最多可以容忍 2 个恶意验证者（拜占庭容错）。

## 第五部分：验证者共识 (Validator Consensus)

### V.1 验证者集 (Validator Set)

Taxoin 使用 **经过许可的验证者集**，共 7 个节点。

```
  VALIDATOR_COUNT = 7
  BYZANTINE_FAULTS = 2       # f = floor((7-1)/3)
  QUORUM_SIZE = 5            # 2f + 1 = 5
```

这意味着：最多 2 个验证者可以是恶意的或有故障的，
网络仍然正常运行。

### V.2 验证者的职责

| 角色 | 描述 |
|------|------------|
| **创世证明 (Genesis attestation)** | 验证新人类（3/7） |
| **服务验证 (Service verification)** | 验证新服务注册 |
| **交易验证 (Transaction validation)** | 检查签名、余额、双重支付 |
| **争议仲裁 (Dispute arbitration)** | 解决双方冲突 |
| **合并共识 (Merge consensus)** | 最终确定状态转换（现有第 4 阶段） |

### V.3 共识协议

Taxoin 实现 Tendermint 风格共识（在第 4 阶段已构建）：

```
  提议 (PROPOSE) → 预投票 (PREVOTE) → 预提交 (PRECOMMIT) → 提交 (COMMIT)
```

每一轮：
1. 一个验证者提出一批经过共同证明的交易
2. 所有 7 个验证者独立验证（5/7 法定人数）
3. 通过验证的批次通过 git 合并到主链中
4. 状态更新（余额、声誉、服务注册表）

Tendermint 共识已完全实现并经过测试
（200 个测试，全部通过）。

### V.4 验证者诚实度

验证者不是匿名的。他们是已知的实体：
- 参与了创世（人性证明）
- 有声誉风险
- 如果恶意，可以由 5/7 投票替换

不诚实的验证者：
1. 失去声誉
2. 可以由 5/7 投票驱逐
3. 替换验证者通过认证

## 第六部分：Git 架构

### VI.1 为什么用 Git？

Taxoin 构建在 Git 之上，因为 Git 是经过验证的 DAG（有向无环图）：

| 区块链 | Git |
|------------|-----|
| 区块 | 提交 |
| 父哈希 | 父提交 SHA |
| 默克尔根 | Git 树 SHA |
| 链 | 分支历史 |
| 分叉 | 分支 |
| 合并 | 合并提交 |
| 轻客户端 | git clone --depth=1 |

### VI.2 工作原理

```
  .taxoin/
  └── .git/
      ├── objects/       # 区块作为 git 提交
      ├── refs/heads/    # 分支 + 主链
      └── NOTES_MERGE/   # 服务注册表、元数据
```

每批交易是一个 git 提交。每个服务注册是一个 git note。
每个状态更改是一个 git 合并。

**不需要数据库。不需要索引。只需要 Git。**

### VI.3 优势

| 属性 | Git 如何提供 |
|----------|-------------------|
| 不可变性 | 基于 SHA 的内容寻址 |
| 历史 | git log — 完整审计跟踪 |
| 分发 | git clone — 几秒钟内完整节点 |
| 轻客户端 | git clone --depth=1 |
| 备份 | git push — 即时备份到任何远程 |
| 验证 | git verify — 签名检查 |
| 分叉 | git branch — 并行实验 |

## 第七部分：代币经济学 (Tokenomics)

### VII.1 总供应量

```
  MAX_GENESIS_SUPPLY   = 21,000,000 Ⓣ  （硬上限）
  MAX_SERVICE_SUPPLY   = 无限          （受实际经济限制）
```

创世供应量上限为 2100 万。服务供应量是无限的，
但服务供应中的每个 Ⓣ 都是**赚来的**，而不是创造的。

### VII.2 创世分配

```
  420,000 人 × 50 Ⓣ = 21,000,000 Ⓣ
  ──────────────────────────────────────
  没有预挖矿。没有 ICO。没有内部人分配。
  每个人都得到完全相同的数量。
```

### VII.3 流通速度

Ⓣ 旨在流通，而不是囤积：

```
  创世:  50 Ⓣ → 花费在服务上
  提供者: 赚取的 Ⓣ → 花费在其他服务上
  消费者: 支付服务费用 → 提供者赚取 → 再次花费
```

Taxoin 不是价值储存手段。它是点对点服务经济的**交换媒介**。

### VII.4 费用

没有协议级别的交易费用。验证者不获得区块奖励（传统意义上没有区块）。

验证者可以收取自愿费用用于：
- 创世证明
- 争议仲裁
- 服务验证

费用市场是自由的：验证者在价格和可靠性上竞争。

## 第八部分：安全模型

### VIII.1 威胁矩阵

| 威胁 | 缓解措施 |
|--------|-----------|
| Sybil 攻击（假人类） | 人性证明 + 3/7 证明 |
| 双重支付 | 冲突检测器（第 2 阶段）+ 支出跟踪 |
| 虚假服务（收钱，不交付） | 共同证明 + 争议解决 |
| 验证者串通（3/7 恶意） | 拜占庭容错 (f=2) |
| 注册表操纵 | Git 不可变性 + 验证者签名 |
| 余额伪造 | 所有交易上的 ECDSA 签名 |
| 重放攻击 | 每个地址的 Nonce 跟踪 |
| 网络分区 | 八卦协议 + 基于超时的活性 |

### VIII.2 信任假设

Taxoin 不是无需信任的。它是**信任最小化的**：

| 你必须信任 | 你不需要信任 |
|---------------|-------------------|
| 5/7 验证者是诚实的 | 任何单个验证者 |
| 你的私钥是安全的 | 网络是"公平的" |
| 验证者正确验证人类 | 矿工（不存在） |
| | 质押者（不存在） |
| | 预言机（证明是相互的） |

## 第十部分：大局观

### X.1 Taxoin 改变了什么

| 时代 | 机制 | 浪费的资源 |
|-----|-----------|----------------|
| **2008 年之前** | 物物交换 / 法定货币 | 信任 |
| **2008 年（比特币）** | 工作量证明 (Proof of Work) | 电力 |
| **2015 年（以太坊）** | 权益证明 (Proof of Stake) | 资本 |
| **2026 年（Taxoin）** | 贡献证明 (Proof of Contribution) | 没有 |

### X.2 最终目标

一个自我维持的去中心化服务经济，其中：

- **任何人都可以通过提供已有的服务来赚取收入**
- **任何人都可以用通过贡献赚取的代币来支付**
- **没有人可以作弊，因为双方都必须证明**
- **没有人被排除在外，因为唯一的要求是成为人类**
- **没有能源被浪费，因为计算服务于人，而不是哈希值**

### X.3 名称由来

**Taxoin** 来自 *taxi*（出租车）+ *coin*（币）。

出租车司机提供真实、有价值的服务——交通。
乘客为该服务付费。双方证明乘车确实发生了。
没有挖矿。没有质押。没有投机。

只有 **服务、支付和证明。**
