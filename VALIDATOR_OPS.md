# Validator Operations: Что и как нажимать

> Практическое руководство для валидатора Taxoin.
> Никакой теории — только кнопки.

---

## 1. Запустить валидатора (один раз)

### Через Docker (рекомендуется)

```bash
# 1. Установить Docker
curl -fsSL https://get.docker.com | sh

# 2. Скачать проект
git clone https://github.com/taxoin/taxoin.git
cd taxoin

# 3. Запустить
docker compose up -d
```

**Готово.** Ваш валидатор работает на порту `47701`.

Проверить:
```bash
curl http://localhost:47701/health
# → {"status": "ok"}
```

### Без Docker

```bash
pip install -r requirements.txt
python3 -m src.api --port 47701
```

## 2. Проверить что валидатор работает (ежедневно)

```bash
# Статус сети
curl http://localhost:47701/api/status
# → { "blocks": 42, "validators": 3, ... }

# Проверить логи
docker compose logs --tail=20 validator-1
```

**Зелёные сигналы:**
- `/health` → `200 OK`
- В логах нет `ERROR`
- Количество блоков растёт

**Красные сигналы:**
- `/health` не отвечает → валидатор упал
- `Merge failed` в логах → проблемы с консенсусом
- Диск заполнен → чистить `.taxoin/`

## 3. Участвовать в консенсусе (автоматически)

**Валидатору не нужно ничего нажимать для консенсуса.**

```python
# consensus.py делает всё сам:
while True:
    propose()    # предложить блок
    prevote()    # голосовать YES/NO
    precommit()  # подписать кворум
    commit()     # записать блок
```

Валидатор проверяет:
1. Подпись отправителя ✅
2. Баланс отправителя ✅
3. Нет double-spend ✅
4. Совпадает ли с другими валидаторами ✅

Голосует автоматически. Человек не нужен.

**Когда человек нужен:**
| Действие | Как | Частота |
|----------|-----|---------|
| **Genesis аттестация** | `taxoin genesis attest <address>` | При появлении нового человека |
| **Dispute** | `taxoin dispute vote <id> <verdict>` | При споре |
| **Обновление софта** | `docker compose pull && docker compose up -d` | При выходе новой версии |
| **Проверка здоровья** | `curl /health` | Раз в день |

## 4. Genesis — подтвердить нового человека (изредка)

Когда новый человек хочет получить genesis-монеты, валидатор должен
подтвердить, что это реальный уникальный человек:

```bash
# 1. Человек пришёл, показал паспорт/видеозвонок
# 2. Валидатор подтверждает:
curl -X POST http://localhost:47701/api/testnet/faucet \
  -H "Content-Type: application/json" \
  -d '{"address": "0xНОВЫЙ_ЧЕЛОВЕК"}'

# 3. Когда 3 из 7 валидоров подтвердили → человек получает genesis
```

**Нужно 3 подтверждения от разных валидоров.**
Каждый делает `curl` или нажимает кнопку в Telegram.

---

## 5. Dispute — разрешить спор (редко)

Если потребитель и поставщик не сошлись во мнении:

```bash
# 1. Посмотреть открытые споры
curl http://localhost:47701/api/disputes

# 2. Проголосовать
curl -X POST http://localhost:47701/api/dispute/vote \
  -H "Content-Type: application/json" \
  -d '{"dispute_id": "abc123", "verdict": "consumer_wins"}'
```

**Нужно 5 из 7 голосов для решения.**

## 7. Что валидатор НЕ должен делать

```
❌ Не выключать ноду без предупреждения
   → Пропущенные блоки → штраф репутации

❌ Не голосовать "не глядя"
   → Подписывать только то, что сам проверил

❌ Не хранить private_key в открытом виде
   → Использовать аппаратный кошелёк или encrypted файл

❌ Не запускать validator и consumer на одной ноде
   → Конфликт интересов
```
