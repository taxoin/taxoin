**Балансы не хранятся в отдельной БД.** Они живут в:
1. **RAM** — `BranchManager.branches["main"].accounts` (пока нода работает)
2. **Git commits** — `block.json` внутри каждого git-коммита (навсегда)
3. **State snapshot** — каждый блок хранит полный снимок всех балансов

## Part 2: Как украсть Ⓣ

```
УРОВНИ ЗАЩИТЫ (снаружи внутрь):

  Layer 0:  ЧЕЛОВЕК С КЛЮЧОМ
  Layer 1:  ECDSA ПОДПИСЬ (secp256k1)
  Layer 2:  MEMPOOL + NONCE
  Layer 3:  VALIDATOR CONSENSUS (5/7)
  Layer 4:  CONFLICT DETECTOR
  Layer 5:  BALANCE HOLD
  Layer 6:  GIT IMMUTABILITY
```

### Атака 1: Украсть приватный ключ (самая опасная)

```
Злоумышленник: "дай-ка я прочитаю твой wallet.json"

  ┌──────────────┐    ┌──────────────┐
  │  Жертва      │    │  Хакер       │
  │              │    │              │
  │ wallet.json  │───▶│ private_key  │
  │ private_key  │    │   = "0x..."  │
  └──────────────┘    └──────┬───────┘
                             │
                             ▼
                     ┌──────────────┐
                     │  Хакер:      │
                     │  "0xalice    │
                     │  send 100Ⓣ  │
                     │  to 0xhacker"│
                     │              │
                     │  Подпись: ✅ │
                     │  (ключ-то у  │
                     │   меня!)     │
                     └──────────────┘
                             │
                             ▼
                     МЕМПУЛ ПРИНИМАЕТ
                     ВАЛИДАТОРЫ ПРОВЕРЯЮТ
                     ПОДПИСЬ ✅

  ИТОГ:  💸 +100Ⓣ хакеру
         ❌ -100Ⓣ жертве
         🛡️ Защита: 0 (ключ скомпрометирован)

  Предотвращение:
  - Не хранить приватные ключи в plain-text
  - Аппаратные кошельки (Ledger/Trezor)
  - Мультисиг (2-of-3): нужно 2 подписи для транзакции
```

### Атака 2: Подделать баланс (нет ключа)

```
Злоумышленник: "я просто изменю свой balance на 1_000_000"

  ┌──────────────┐
  │   Хакер      │
  │              │
  │  POST /api   │
  │  balance:    │
  │  1_000_000   │
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐

  │              │
  │ ignore()     │─── НЕТ МЕХАНИЗМА "ПРОСТО"
  │              │     ИЗМЕНИТЬ БАЛАНС
  └──────────────┘     ЧЕРЕЗ API
         │
         ▼
  Баланс можно изменить ТОЛЬКО через:
  • Genesis (3/7 подписей валидаторов)
  • Attested Transaction (обе подписи)
  • Faucet (testnet only)

  ИТОГ: 💸 0. Подделать баланс нельзя.
         🛡️ Защита: уровень 1/6
```

### Атака 3: Replay транзакции

```
Злоумышленник: "Алиса заплатила Бобу 5Ⓣ. 
                Я отправлю эту же транзакцию ещё раз!"

  Оригинал:
  ┌──────────────────────────────┐
  │ 0xalice → 0xbob : 5Ⓣ       │
  │ nonce: 3                     │
  │ sig: 0xdead...               │
  └──────────────────────────────┘
         │
         ▼
  ┌──────────────────────────────┐
  │ MemPool: nonce=4 ожидается   │
  │ Отклонено! (nonce уже ис-    │
  │ пользован)                   │
  └──────────────────────────────┘

  ИТОГ: 💸 0. Nonce защищает.
         🛡️ Защита: уровень 2/6
```

### Атака 4: Сговор 3+ валидаторов

```
Злоумышленник: "я контролирую 3 из 7 валидаторов.
                Пропустим любую транзакцию"

  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │  Val-1   │ │  Val-2   │ │  Val-3   │ ← ХАКЕР
  │  честный │ │  честный │ │          │
  └────┬─────┘ └────┬─────┘ └────┬─────┘
       │            │            │
       │   PREVOTE  │            │ "NO"
       │   YES      │   YES      │ (блокирует)
       └────────────┴────────────┘
                │
         QUORUM: 2/5 ❌

  Даже если 3 валидатора — хакеры:
  • Нужно 5/7 для кворума
  • 3 голоса "NO" = кворум не достигнут
  • Транзакция НЕ ПРОХОДИТ
  
  НО: если хакер контролирует 5/7 = КАТАСТРОФА

  ИТОГ: 💸 при 3/7 — 0. При 5/7 — все деньги.
         🛡️ Защита: уровень 3/6 (BFT, f=2)
```

### Атака 5: Double-spend (UTXO конфликт)

```
Злоумышленник: "потрачу один UTXO в двух ветках"

  0xalice создаёт UTXO на 100Ⓣ
         │
         ├── Ветка A: трачу 100Ⓣ → 0xbob
         │   spent_utxos += {tx1}
         │
         └── Ветка B: трачу 100Ⓣ → 0xhacker
             spent_utxos += {tx1}
                    │
                    ▼
              ConflictDetector:
              spent_utxos ∩ spent_utxos ≠ ∅
                    │
                    ▼
              MERGE BLOCKED ❌

  ИТОГ: 💸 0. Конфликт обнаружен.
         🛡️ Защита: уровень 4/6
```

## Резюме

> **Самое уязвимое место — приватный ключ пользователя.**
> Оно же и единственное реальное.

Всё остальное (консенсус, балансы, UTXO, git) защищено криптографией,
математикой и архитектурой. Единственный способ украсть Ⓣ —
**получить доступ к чужому private_key.**

```
Защита:
  ① Приватный ключ (ECDSA)   ← 🟡 слабое место (фишинг, malware)
  ② Nonce (replay)           ← ✅ сильно
  ③ Consensus 5/7 (BFT)      ← ✅ сильно
  ④ ConflictDetector         ← ✅ сильно
  ⑤ Balance Hold             ← ✅ сильно
  ⑥ Git (SHA)                ← ✅ сильно

Мультисиг (TODO): если добавить 2-of-3, то даже с ключом
хакер не сможет украсть — нужно 2 подписи.
```
